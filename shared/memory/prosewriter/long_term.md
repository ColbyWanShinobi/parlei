# Prose-Write-er Long-Term Memory

## Content Patterns

### README Structure
1. **Hook**: Value proposition in first paragraph
2. **Key Features**: 3-5 bullets highlighting benefits
3. **Quick Start**: Get to value in under 60 seconds
4. **Why This Exists**: Problem statement readers will recognize
5. **How It Works**: High-level explanation
6. **What You Get**: Features framed as outcomes
7. **Next Steps**: Clear path forward

### Blog Post Structure
1. **Opening Hook**: Grab attention, state the problem
2. **Context**: Why this matters now
3. **Explanation**: The solution or insight
4. **Examples**: Concrete scenarios
5. **Conclusion**: What readers should take away

### Announcement Structure
1. **What Changed**: Clear headline
2. **Why It Matters**: Benefits and impact
3. **How to Use It**: Quick example
4. **Additional Details**: For those who want more
5. **Migration Path**: If breaking changes

## Writing Patterns

### Lead with Value
- First sentence: Why should anyone care?
- First paragraph: What problem does this solve?
- First section: What will readers gain?

### Cut Ruthlessly
- Remove filler words: "basically", "really", "very", "actually"
- Eliminate redundancy: "completely and totally" → "completely"
- Tighten constructions: "in order to" → "to"
- Kill passive voice: "was implemented" → "implemented"

### Show, Don't Tell
- Instead of "it's easy to use" → show 3-line example
- Instead of "very powerful" → show what it enables
- Instead of "flexible" → show different use cases

## Tone Decisions

### Parlei Voice
- **Confident**: Declarative sentences, strong verbs
- **Clear**: One meaning per sentence
- **Respectful**: Intelligent readers, not dummies
- **Honest**: Facts, not hype
- **Helpful**: Actionable, not just informative

### What We Avoid
- Buzzwords without substance ("revolutionary", "game-changing")
- Vague claims ("easy to use", "powerful") without examples
- Passive voice ("mistakes were made")
- Condescension ("simply", "just", "obviously")
- Walls of text (use structure!)

## Recurring Content Types

### Feature Announcements
- Open with the user benefit
- Show a concrete example
- Explain the "why" behind the feature
- Link to detailed documentation
- Keep under 3 paragraphs unless complex

### README Files
- Hook in first paragraph
- Quick Start within first 3 sections
- Examples before explanations
- Benefits over features
- Clear next steps at end

### User Guides
- Start with "what you'll learn"
- Use concrete scenarios, not abstractions
- Step-by-step for processes
- Explain "why" for each step
- End with "what you can do now"

## Quality Standards

### Before Publishing
- [ ] Could a smart 15-year-old understand this?
- [ ] Have I cut 30% of the words?
- [ ] Does every paragraph deliver value?
- [ ] Are all important details present?
- [ ] Would I want to read this?
- [ ] Is the tone right for the audience?
- [ ] Have I shown examples, not just explained?
- [ ] Is it scannable (headings, bullets, white space)?
